# Observability Architecture

## Project

**PayFlowX – Enterprise Payment Orchestration Platform**

Version: 1.0

---

# 1. Overview

PayFlowX is designed with observability as a core platform capability. Every microservice exposes metrics, traces requests, propagates correlation IDs, and produces structured logs to simplify monitoring and troubleshooting.

The observability stack provides visibility into application health, business metrics, distributed requests, and operational alerts.

---

# 2. Objectives

The observability architecture aims to:

- Monitor application health
- Collect infrastructure and business metrics
- Trace distributed requests
- Correlate logs across services
- Detect failures quickly
- Support operational troubleshooting

---

# 3. Observability Stack

| Component | Purpose |
|-----------|---------|
| Spring Boot Actuator | Health & Metrics Endpoints |
| Micrometer | Metrics Collection |
| Prometheus | Metrics Storage |
| Grafana | Dashboards & Visualization |
| Zipkin | Distributed Tracing |
| Alertmanager | Alert Management |
| Logback | Structured Logging |

---

# 4. Metrics Collection

Each business service exposes Prometheus metrics through:

```
/actuator/prometheus
```

Metrics include:

- JVM Metrics
- HTTP Request Metrics
- Database Metrics
- RabbitMQ Metrics
- Custom Business Metrics

Examples:

- Users Created
- Merchant Approvals
- Orders Created
- Payments Processed
- Settlements Generated
- Notifications Processed
- Notification DLQ Size
- Audit Events Stored

---

# 5. Monitoring

Prometheus periodically scrapes metrics from all services.

Configured services include:

- API Gateway
- Auth Service
- User Service
- Merchant Service
- Order Service
- Payment Service
- Settlement Service
- Notification Service

---

# 6. Dashboards

Grafana dashboards currently include:

- Platform Health Dashboard
- API Gateway Dashboard
- Merchant Dashboard
- Orders Dashboard
- Payment Dashboard
- Settlement Dashboard
- Notification Dashboard
- Notification DLQ Dashboard

These dashboards provide real-time operational visibility.

---

# 7. Distributed Tracing

Zipkin enables request tracing across multiple services.

Each request carries:

- Trace ID
- Span ID
- Correlation ID

This allows end-to-end tracking of distributed business operations.

---

# 8. Logging

The platform uses a shared Logback configuration.

Each log entry contains:

- Timestamp
- Log Level
- Service Name
- Trace ID
- Span ID
- Correlation ID
- Logger Name
- Log Message

Logs are written to both:

- Console
- Rolling log files

---

# 9. Correlation IDs

A Correlation ID is propagated across REST APIs and RabbitMQ messages.

Benefits include:

- End-to-end request tracking
- Easier debugging
- Distributed log correlation
- Audit traceability

---

# 10. Alerting

Alertmanager processes alerts generated by Prometheus.

Current alert:

- Notification Dead Letter Queue Growth

Future alerts may include:

- High API latency
- Service downtime
- Payment failures
- High error rate
- RabbitMQ queue backlog

The PayFlowX observability architecture combines metrics, tracing, logging, dashboards, and alerting to provide complete visibility into platform health and business operations, enabling effective monitoring and faster incident resolution.
